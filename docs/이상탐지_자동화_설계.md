# 상담·트래픽 이상탐지 자동화 — 설계

홈페이지 상담과 방문 트래픽에서 **가짜/봇/스팸/도용**을 자동으로 찾아내고, 서로 대조해
"광고비를 갉아먹는 이상 유입"을 걸러내기 위한 설계 문서.

> 전제: 상담메일이 `consult_raw` 테이블에 계속 쌓인다(네이버→지메일 자동전달→`collect_mail`).
> 이 문서는 **그 데이터가 쌓인다는 가정 하에, 무엇을·어떻게 자동으로 검사·대조·판정할지**를 미리 정리한 것이다.
> (값·개인정보는 여기 적지 않는다. 규칙과 구조만.)

---

## 0. 한 장 요약

두 개의 축을 자동으로 검사하고, 시각으로 서로 대조한다.

```
  [축 A] 이상 상담(consult_raw)          [축 B] 이상 IP·방문(visit_log)
   같은번호·다른이름 / 반복번호            한 IP 다수제출 / 데이터센터·VPN 대역
   무효번호 / 이름이상 / 중복템플릿          광고클릭 없이 유입 / 봇 UA / 새벽버스트
        │                                        │
        └──────────── [축 C] 시각 대조 ───────────┘
              상담 제출시각 ↔ 방문시각 매칭 → "이 상담을 낸 IP" 추정
              → 스팸상담↔IP 서로 위험도 가중 → 차단후보 산출 → 광고낭비 추정
```

산출물: ① 대시보드 '상담 품질·보안' 탭(의심 상담·IP·차단목록) ② 일일/주간 위험요약 리포트
③ 차단목록(전화·이름=`consult_block`, IP=.htaccess 생성물).

---

## 1. 데이터 재료

| 테이블 | 내용 | 상태 | 자동화 |
|---|---|---|---|
| `consult_raw` | 상담메일(이름·전화·지역·상담내용·시각·센터·분류) | 🟡 소스 확정 대기(지메일 전달) | `collect_mail`(collect_all) |
| `visit_log` | 홈페이지 방문(IP·시각·접속경로·OS·기기·UA·유입매체) | 🔴 **미자동화**(현재 1회성 스크랩) | **Phase 1에서 신설 필요** |
| `inq` | 통합문의(이름·키워드·분류·상담·수임) | ✅ | `sync_inq` |
| `ad_keyword`/`ad_etc` | 매체별 광고비·클릭 | ✅ | collect_all |
| `consult_block` | 상담 차단목록(번호/이름) | ✅ | 대시보드 저장 |

핵심 공백: **`visit_log` 자동적재**. `scrape_admin.py`가 홈페이지 관리자 방문로그를 긁는 기계는 있으나
현재는 수동 1회성. IP 축을 자동화하려면 이걸 매일 BQ에 적재하는 스케줄이 먼저다.

## 2. 축 A — 이상 상담 규칙 (consult_raw)

각 상담 1건에 대해 아래 규칙을 검사, 걸리면 사유 태그를 붙이고 점수를 더한다.
(대부분 대시보드 `render_spam`에 이미 구현 — 규칙으로 정식화)

| 규칙 | 신호 | 판정 | 가중치 |
|---|---|---|---|
| 같은 번호·다른 이름 | 한 번호로 이름 2개+ 제출 | 봇/도용 강함 | ★★★ |
| 반복 번호 | 같은 번호 ≥3회 | 반복 제출 | ★★ |
| 무효/비정상 번호 | `01[016789]\d{7,8}` 형식 아님(국제·0000·문자) | 가짜 번호 | ★★ |
| 이름 이상 | 기호만·이니셜·업체/직함성·숫자·SQL/코드(`select`,`union`,링크) | 가짜/봇 | ★★ |
| 중복 템플릿 | 상담내용 글자 동일 + **서로 다른 번호 2개+** | 대량 봇 스팸 | ★★★ |
| 분당 버스트 | 같은 분(YYYY-MM-DD HH:MM)에 ≥3건 제출 | 자동제출 | ★★ |
| 새벽 집중 | 00~05시 비중 급증(개별보다 집계 신호) | 봇 의심 | ★ |
| 무의미 내용 | 초단문(공백제거 <5자)·외국어 스팸문구·링크 | 저품질 | ★ |

> 과거 실데이터(상담 3,829건)에서 이 규칙들로 이미 확인된 것: 같은번호·다른이름 19, 반복번호 118,
> 무효번호 669, 중복템플릿 9종(해외 백링크 봇), 분당버스트 84개 분 등. → 규칙이 실제로 먹힌다.

---

## 3. 축 B — 이상 IP·방문 규칙 (visit_log)

방문 1건/IP 1개에 대해 검사. (과거 `classify_ips.py`·`riskgroup.py`·`full_classify.py`에서 프로토타입한 로직을 규칙화)

| 규칙 | 신호 | 판정 | 가중치 |
|---|---|---|---|
| 한 IP 다수 상담 | 같은 IP가 짧은 기간 상담 여러 건 | 반복 유입 | ★★★ |
| 데이터센터/VPN/호스팅 대역 | ASN이 클라우드·호스팅(사람 아님) | 봇 인프라 | ★★★ |
| 광고 클릭 없이 유입 | referrer/utm 없이 직접·이상 경로 | 비정상 유입 | ★★ |
| 봇 UA | 크롤러/헤드리스 User-Agent | 자동 접근 | ★★ (단, 검색엔진 정상봇은 화이트리스트) |
| 새벽 버스트 방문 | 특정 IP가 새벽 반복 접속 | 봇 | ★ |
| 비정상 접속경로 | 존재하지 않는 파라미터·반복 요청 | 스캐닝 | ★ |
| 해외 IP | 한국 서비스인데 해외 대역(문맥 따라) | 참고신호 | ★ |

> 크롤러는 **차단하면 안 되는 정상봇(구글·네이버 검색)** 과 **가짜 트래픽 봇**을 반드시 구분.
> 과거 작업에서 '차단금지 크롤러' 목록을 이미 뽑아둠 → 화이트리스트로 재사용.

---

## 4. 축 C — 상담 ↔ 방문 시각 대조 (교차검증)

두 축을 **시각(datetime)** 으로 잇는다. (task #3에서 프로토타입한 시각대조를 자동화)

1. 상담 제출시각(`consult_raw.datetime`) 근처(±N초/분) 방문로그(`visit_log`)를 매칭 → **그 상담을 낸 IP 후보** 추정.
2. **양방향 가중**:
   - 스팸으로 판정된 상담 → 매칭된 IP의 위험도 +.
   - 이미 위험한 IP → 그 IP가 낸 상담의 위험도 +.
3. 수렴: 여러 번 서로 강화되며 "확실한 가짜 유입 클러스터"가 도드라진다.
4. **광고 낭비 추정**: 가짜 상담·IP가 특정 캠페인/키워드/매체/랜딩에 몰리는지 → `ad_keyword`/유입매체와 대조.
   → "이 캠페인은 클릭 대비 진짜 상담이 거의 없다" 같은 낭비 신호.

주의: 시각 매칭은 **추정**이다(같은 분 여러 방문이면 모호). 단독 근거로 차단하지 말고 다른 규칙과 합산 판정.

## 5. 점수화 & 판정

- 상담·IP 각각 규칙 히트의 가중치를 합산 → **위험 점수**.
- 임계값 이상 = '의심', 상위 = '차단 후보'. (임계값은 실데이터 보고 튜닝)
- 판정은 **자동 표시**까지. 실제 차단은 사람이 확인 후:
  - 전화·이름 → 대시보드에서 `consult_block`에 체크 저장(이미 구현) → 표에 `차단됨` 배지.
  - IP → `.htaccess` deny 목록 파일 생성(과거 산출물 형식) → 홈페이지 업체에 전달.

## 6. 자동화 파이프라인 (어떻게 굴러가나)

```
매일:
  collect_all ─┬─ collect_mail   → consult_raw   (상담)
               └─ (신설) scrape_visits → visit_log (방문)   ← Phase 1
      │
  (신설) detect_anomalies 배치 → 규칙 적용·시각대조·점수 → consult_flags / ip_flags 스냅샷  ← Phase 2
      │
  대시보드(실시간 계산) + 일일 위험요약 리포트(급증 시 GitHub 이슈/메일)  ← Phase 3
```

- 탐지 계산은 두 가지 방식 병행 가능:
  - **실시간**(대시보드 열 때 계산) — 지금 `render_spam` 방식. 가볍고 즉시.
  - **일일 배치 스냅샷**(`*_flags` 테이블) — 리포트·추세·알림용. 무거운 시각대조는 여기서.
- BigQuery 무료티어 규칙 준수: 전부 **SELECT + WRITE_TRUNCATE 적재**, DML 금지.

## 7. 대시보드 반영 (상담 품질·보안 탭)

- 현재: 축 A(상담) + 차단목록 = 구현 완료(`consult_raw` 채워지면 자동 ON).
- 추가 예정:
  - 축 B(이상 IP) 표: 위험 IP·사유·방문수·매칭상담수 + `.htaccess` 내려받기 버튼.
  - 축 C(대조) 뷰: "이 IP ↔ 이 상담들" 묶음, 캠페인별 가짜상담 비율.
  - 상단 KPI에 '오늘 신규 의심 상담/IP', 추세 스파크라인.

## 8. 단계별 로드맵

| 단계 | 내용 | 선행조건 |
|---|---|---|
| **Phase 0 (완료)** | consult_raw 채워지면 상담보안 탭 자동 ON(축 A + 차단목록) | 지메일 전달 세팅 |
| **Phase 1** | `visit_log` 자동적재(`scrape_visits`→BQ, 매일). 스키마·중복방지 | 홈페이지 관리자 로그 접근 안정성 |
| **Phase 2** | 축 B 규칙 + 축 C 시각대조 자동화 → `ip_flags`/대조 뷰, 대시보드 IP 탭 | Phase 1 |
| **Phase 3** | 일일 위험요약 리포트(급증 알림) + `.htaccess`·차단목록 자동생성 | Phase 2 |
| **Phase 4** | 광고 낭비 추정(가짜상담·IP ↔ 캠페인/키워드/매체) 연계 리포트 | Phase 2, 광고데이터(있음) |

## 9. 열린 질문 · 의존성

- **상담 소스 확정**: 홈페이지가 개인메일로도 직접 쏠 수 있는지 업체 확인 중.
  - 직접 수신 가능 → 지메일에서 바로 수집(전달 불필요, 더 깔끔·과거이력도 유리).
  - 불가 → 네이버→지메일 자동전달 유지.
  - 어느 쪽이든 `collect_mail`은 메일서버 무관(`MAIL_*` env)이라 그대로 동작.
- **방문로그 원천**: `scrape_admin.py`의 관리자 로그 접근이 계속 되는지(로그인·세션 만료·차단 위험). 안 되면 IP 축은 대안 필요(예: 광고 랜딩단 로깅, GA4 보조).
- **과거 이력 백필**: 전달은 소급 안 됨 → 지메일 경로는 '오늘 이후'만 쌓임. 과거 3,829건을 대시보드에 함께 보려면 별도 1회 백필 검토(개인정보 취급 경로 확정 필요).
- **IP 정확도**: 상담을 낸 실제 IP를 방문로그에서 얼마나 신뢰도 있게 잇느냐가 축 C의 관건.
