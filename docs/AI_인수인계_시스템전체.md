## 1. ⭐ 데이터 연결 모델 (가장 중요 — 이걸 이해해야 함)

### 핵심 원칙
**개별 건을 1:1로 잇지 않는다. "카테고리(사건유형)"를 공통 키로 삼아 합계 수준에서 잇는다.**

```
키워드 ──속함──▶ 캠페인 ──정규화──▶ [카테고리] ◀──별칭통일── 사건유형 ◀── 계약
(ad_keyword)     (media별 명명규칙)  _campaign_to_category()  CAT_ALIAS      (계약시트)
                                          ▲
                                          │ 카테고리 컬럼
                                       문의(inq) ── 검색키워드(기입률 낮아 신뢰도↓)
```

공통 키 = **카테고리**: 메인·형사·성범죄·금융범죄·건설부동산·학교폭력·음주·교통·이혼가사·소년범죄·행정소송·회생파산·외국인 등.
광고도, 문의도, 계약도 전부 이 카테고리로 환산해야 서로 만난다.

### 두 축은 절대 섞지 않는다
- **축1 = 광고 성과(단위 '건')**: 캠페인/키워드 → 문의 → 상담 → 수임
- **축2 = 사건 매출(단위 '원')**: 사건유형 → 계약금액 → 입금 → 미수

둘은 건별로 직접 안 이어지고 **카테고리에서 합계로만 비교**. 그래서:
- **ROAS = 카테고리별 광고비 ÷ 카테고리별 매출.** 반드시 **계약 기준 / 입금 기준**을 구분(미수율 높아 둘이 크게 다름, 계약≫입금).
- **월 단위 이상**으로 해석(문의→수임→입금 시차 큼. 일 단위 ROAS·효율 판단 금지).

### 각 연결점이 코드에서 일어나는 곳
| 연결 | 어디서 | 규칙 |
|---|---|---|
| 키워드 → 캠페인 | `ad_keyword`(media·campaign·keyword) | 키워드는 캠페인 하위. 네이버 키워드ID→이름은 `naver_kw_master` |
| 캠페인 → 카테고리 | `_campaign_to_category(name)` | 접두/날짜접두/시간대·신규 접미/(삭제)꼬리 제거 + `CAT_ALIAS` 별칭통일 |
| 문의 → 카테고리 | 문의시트 `카테고리` 컬럼 | 검색키워드 컬럼도 있으나 "미확인" 다수 → 키워드 전환분석 표본 왜곡 |
| 계약 → 사건유형(=카테고리) | 계약시트 사건유형 explode | 여기도 `CAT_ALIAS`로 광고 카테고리와 이름 맞춤 |

### 캠페인 명명 규칙 (정규화가 필요한 이유)
- **네이버**: `A.메인_1724` = `A.`(정렬용 접두, 무시) + `메인`(카테고리) + `_1724`(시간대 접미). **시간대 캠페인(_1724,_1117,_항시…)을 합산**해야 카테고리 성과가 온전. 예: `OFF.일반형사_1117(삭제)`→`형사`, `성범죄_항시_신규`→`성범죄`.
- **구글**: `250728_성범죄` = 날짜접두(`250728_`) 제거 → `성범죄`.
- **네이버 캠페인 ID 문제**: `collect_naver`가 campaign 컬럼에 캠페인 ID(`cmp-…`)를 저장하는 경우가 있어(이름 매핑 마스터 없음), `_naver_campaign_namemap()`이 `ad_budget` 스냅샷에서 최신 캠페인명을 복원한 뒤 정규화한다. (2026-06 캠페인 재생성 후 분류 깨지던 것 복원용.)

`_campaign_to_category()` 실제 처리 순서:
1. `cmp-`로 시작하면 `ad_budget`으로 이름 복원
2. `(…)` 꼬리 괄호 제거 → 3. `^[A-Za-z]+\.` 정렬접두 제거 → 4. `^\d{4,6}_` 구글 날짜접두 제거
5. `_숫자|_항시|_상시|_신규` 접미 제거(이중 접미 대응 2회) → 6. `CAT_ALIAS` 별칭 적용

```python
CAT_ALIAS = {"일반형사":"형사","음주운전":"음주","외국인/출입국":"외국인",
             "교통사고":"교통","하자/보수":"하자보수","의료분쟁":"의료","학폭":"학교폭력"}
GOOGLE_CAT_MAP = {"검색광고":"구글메인","성범죄":"구글성범죄","부동산센터":"구글부동산",
                  "금융":"구글금융","형사":"구글형사","음주":"구글음주","학폭":"구글학폭"}
```

## 4. 수집 자동화 스케줄 (KST)

**철학**: GitHub 무료 cron은 몇 시간씩 밀림 → 정시 보장은 **Claude 루틴(CCR 트리거)**이 workflow_dispatch로 즉시 실행. GitHub cron은 백업 1개.

| 시각 | 하는 일 | 트리거 |
|---|---|---|
| 06:30 | collect_all (광고 안전망) | GitHub cron (`30 21 * * *` UTC) |
| **10:00** | collect_all (문의+광고) | Claude 루틴 |
| **11:30** | collect_all | Claude 루틴 |
| **14:00** | collect_all | Claude 루틴 |
| **14:30** | 마감 점검(누락 있으면 collect_all 재실행+보고) | Claude 루틴(이 세션에 보고) |
| 3시간마다 | check_landing 점검 | Claude 루틴 |
| 매시(밀림) | check_landing | GitHub cron |

- **왜 3회(10:00/11:30/14:00)?** KB 직원이 아침 출근해 상담·문의를 **수기로 시트에 올림**(주말·공휴일 다음날 몰림). 그 입력을 오전 회차가 잡는다. collect_all에 sync_inq가 포함되므로 이게 곧 "문의 동기화 3회".
- **계약은 여기 없음** — 라이브(열면 최신).
- **동시쓰기 방지**: 네이버·구글 수집은 `concurrency: bq-ad-keyword-write`로 상호배제.

## 5. ⚠️ 반드시 지킬 운영 규칙 (어기면 데이터 깨짐)

1. **BigQuery 무료티어 → DML(INSERT/UPDATE/DELETE/MERGE) 금지.** 수집은 전부 "전체 읽기 → 자기 media/날짜 제거 → concat → WRITE_TRUNCATE" 멱등 방식. 콘솔 직접 UPDATE/DELETE 금지. AI SQL 도구(`run_safe_sql`, `bq_query.yml`)도 SELECT만.
2. **오늘 데이터는 항상 미수집(어제까지만).** 오늘 수치는 `ad_budget`(실시간 배지)로만 본다.
3. **네이버 브랜드검색(월정액)은 키워드 데이터에 없다** → 메인 등 광고비가 실제보다 적게 잡힘(화면 경고 캡션 있음).
4. **구글 광고비는 VAT 제외**(cost_micros 그대로) → 시트(VAT 포함)와 최대 10% 차이.
5. **적재 범위**: 네이버 키워드 일별 2024-07~(2024-04~06은 `keyword='(월 합계)'` 총액만). 구글 2025-02 중순~. GA4 2026-06-29~.
6. **캠페인 정규화**는 `_campaign_to_category()` 담당(위 §1). 시간대 캠페인 합산 필수.
7. **두 축(건 vs 원) 섞지 말 것**(§1).
8. **ROAS 계약기준/입금기준 구분**(§1). `roas_card(..., paid=...)`에 실입금액 넘기면 입금기준 ROAS·미수액 함께 표시.
9. **수임·입금은 문의와 시차** → 월 단위 이상 해석.

## 6. 시트 구조 의존성 (바꾸면 화면이 조용히 빈다)

app.py는 시트의 **탭 이름·헤더 텍스트**에 부분문자열 매칭으로 의존. 바꾸면 해당 화면이 깨지거나 빈다:
- 탭 이름: `통합문의`, `연간요약`, `기타매체`, `네이버연령`, `네이버성별`, `네이버매체디바이스`, 문의 월별탭 `YY.MM`
- 계약 시트 헤더: `기본보수`(금액)·`입금`·`미수`·`계약유형`·`세부분류/온라인`(신건 판별)·`계약일`·`위임/의뢰인`
- 문의 시트: A열 `1` 플래그, `문의일자`·`이름`·`검색키워드`·`카테고리`·`상담`·`수임` 헤더
- 기타매체 시트: `카카오모먼트/모비온/메타` + `비용/노출/클릭` 접미
- GA4: 데이터셋 `analytics_457680288`, 전환 이벤트명 LIKE 패턴

계약/문의 로더는 시트 장애·헤더 변경 시 **예외 대신 빈 DF 반환**하도록 방어(화면 통째로 안 깨짐).

## 7. AI 기능

- **탭 상단 인사이트 배너**: `ai_insight()` — Haiku(`MODEL_INSIGHT`), 저렴·30분 캐시.
- **AI 질의 탭**: `ai_chat_answer()` — Sonnet(`MODEL_CHAT`). `build_data_context()` 요약 + 필요시 `run_safe_sql`로 `ad_keyword`를 직접 SELECT(읽기전용·화이트리스트·500MB 상한).
- **QnA 원고 생성 탭**: 실수요 키워드 기반 Q&A 원고 생성(Haiku)·법령검증·게시. 생성분은 `qna_draft`에 저장(재접속 복원), 게시분은 `qna_posts`. '성과 보기' 토글은 GA4 유입 기준.
- 모델 ID는 app.py 상단 상수 + `log_ai_usage` 단가표에 하드코딩 → 교체시 두 곳 함께 갱신.
- **AI 분석 패키지**: `build_export_zip()`가 README+long-format CSV ZIP 생성(다른 AI에 올려 분석용).

## 9. 현재 진행/미완 (2026-07 기준)

- **상담메일 수집(consult_raw) 소스 확정 대기**: 네이버 직접 IMAP은 보호조치로 로그인 거부(`Authentication failed`). 앱은 **네이버→지메일 자동전달 → 지메일 IMAP** 경로로 전환 중(`collect_mail.py`는 메일서버 무관하게 일반화 완료: `MAIL_IMAP_HOST/MAIL_ID/MAIL_PW/MAIL_FOLDER`). 홈페이지 폼이 개인메일로 직접 발송 가능한지 업체 확인 중. 소스 확정되면 '상담 품질·보안' 탭이 전화·내용 기준 스팸탐지(같은번호·다른이름, 반복번호, 중복템플릿, 차단목록)로 자동 전환.
- **이상탐지 자동화 로드맵**(`docs/이상탐지_자동화_설계.md`): 상담(축A)·IP(축B)·시각대조(축C) 자동 탐지. Phase1 = `visit_log` 자동적재(현재 방문로그 IP 데이터는 1회성 스크랩만 있음, 자동화 공백). Phase2~4 = IP 규칙·대조·리포트·광고낭비 추정.

## 10. 개발 관례

- 파이썬 3.11. 의존성: `requirements.txt`(app.py) / 각 워크플로 yml의 pip(수집).
- 문법 검증: `python3 -m py_compile app.py`.
- 코드 스타일: **한국어 주석·docstring 유지**, 기존 톤(차분·정직). 색/폰트는 app.py 상단 CSS 토큰(`GOLD/TXT/MUTED/CORAL/LINE` 등, 검정+금색/블루 럭셔리 컨셉).
- **브랜치 규칙**: main 직접 push 금지. `git fetch origin main && git checkout -B <branch> origin/main` → 작업 → PR → CI(py_compile+YAML) 통과 → squash 병합.
- **비밀값 절대 코드/문서/로그에 넣지 않기.** GitHub Secrets / Streamlit Secrets에만.
- 개인정보(상담 이름·전화·내용)는 비공개 BQ에만, 수집 로그엔 집계 수치만.
